# gzhxz 公众号视觉叙述工作流

> 目标：把 AI/科技领域的一手信号，变成编辑能直接使用的图文证据稿。

这个文档是 `.claude/skills/gzhxz-visual-story/SKILL.md` 的人类可读版。

Skill 文件负责让 Claude 执行；这个文档负责让人一眼看懂流程。

---

## 一句话流程

**发现信号 → 判断值不值得写 → 核验事实 → 找核心冲突 → 规划图片 → 写成可发布文章包。**

---

## 阶段 0：Preflight

先别急着写。

开始前要确认：

- 用户是否给了具体主题 / 链接 / GitHub repo / 论文 / 产品发布？
- 如果没有，就先跑选题发现。
- 是否有过往重复选题？
- 是否能找到一手信源？
- 这个话题是否足够新？
- 这个话题是否有视觉化空间？

输出：

```md
## Preflight
- 输入对象：...
- 模式：空白选题 / 指定链接 / 指定项目 / 指定论文
- 初步判断：可继续 / 需要降级 / 不建议写
- 主要风险：...
```

---

## 阶段 1：Topic Engine

如果没有指定题目，就从外部一手源找候选。

优先来源：

- 官方 Blog / Docs / Changelog
- GitHub repos / releases / issues / PRs
- arXiv / lab pages / conference pages
- Product Hunt
- Hacker News
- 研究者、创始人、维护者的公开发布

不建议把中文二手媒体当作选题源头。

它们可以证明“大家在传”，但不能证明“事实是什么”。

---

## 阶段 2：Freshness Gate

判断这个话题是不是还热乎。

推荐标准：

| 等级 | 新鲜度要求 | 说明 |
|---|---|---|
| S | 48–72 小时内有一手更新 | 可以单独成稿 |
| A | 有价值但不够新，或热度/信源略弱 | 可写，可合并 |
| B | 观察线索 | 暂不建议单独写 |
| Drop | 信源弱、重复、事实不清 | 不写 |

注意：

**老话题不是不能写，但不能假装它是今天刚发生的事。**

---

## 阶段 3：Three-layer Source Pack

每个选题至少拆三层信源。

### Layer 1：Primary Evidence

证明“事实是什么”。

例子：

- 官方公告
- 官方文档
- GitHub release
- arXiv paper
- 产品 Changelog
- 维护者原帖

### Layer 2：Reality Layer

证明“真实世界反馈如何”。

例子：

- GitHub issues
- Hacker News comments
- Product Hunt comments
- Reddit / X / 公开社区反馈
- 用户吐槽、疑问、bug、限制

### Layer 3：Background / Comparison

证明“它放在行业里意味着什么”。

例子：

- 竞品
- 历史版本
- 同类项目
- 旧方案的失败点
- 相关技术解释

---

## 阶段 4：Narrative Conflict

新闻不是文章。

文章需要一个冲突。

常见冲突类型：

| 表层新闻 | 真正可写的冲突 |
|---|---|
| 又一个 AI 工具发布 | 它解决的是 demo 到生产之间的哪道坎？ |
| 某模型刷榜 | 榜单指标和真实可用性是否一致？ |
| GitHub 项目突然火了 | 开发者到底在为什么买单？ |
| 产品宣布支持 Agent | 它是真 Agent，还是自动化脚本换皮？ |
| 论文提出新方法 | 它解决了旧方法的什么痛点，又引入什么代价？ |

输出模板：

```md
## Narrative Conflict
- 表层新闻：...
- 真正冲突：...
- 不是 X，而是 Y：...
- 为什么现在重要：...
- 读者能获得什么：...
- 风险边界：...
```

---

## 阶段 5：Visual Storyboard

先规划图，再写正文。

因为公众号图文不是纯文字报告。

推荐图片结构：

| 图 | 作用 |
|---|---|
| 封面图 | 一眼看懂主题和冲突 |
| 一手信源截图 | 证明核心事实 |
| 机制图 | 解释它怎么工作 |
| 对比图 | 说明它和旧方案/竞品不同 |
| 用户反馈截图 | 展示现实层面的疑问、bug 或热度 |
| 时间线 / 架构图 / 评分卡 | 辅助理解复杂信息 |

图片原则：

- 不伪造截图
- 不用看不清的小字图
- 不让图片只是装饰
- 生成图不能冒充证据图
- 每张图都要能回答：它证明了什么？

---

## 阶段 6：Article Draft

正文写作顺序：

1. 先给最强结论
2. 解释为什么现在值得看
3. 拆机制
4. 放证据图
5. 放用户现实反馈
6. 给判断和风险边界
7. 收一个明确结论

风格建议：

- 中文，高密度，短段落
- 先 punchline，再补背景
- 避免“随着 AI 的发展”这种废话开头
- 事实、推测、观点分开
- 不把二手传播写成事实
- 不夸大 demo 能力

---

## 阶段 7：Reviewer Gate

写完后不要马上交。

检查：

### 事实检查

- 核心事实是否都有一手源？
- 日期是否写清楚？
- 是否混淆事实和推测？
- 是否把二手媒体当事实源？
- 是否有无法验证的说法？

### 视觉检查

- 截图是否清晰？
- 来源和日期是否可见？
- 每张图是否有明确用途？
- 生成图是否被误当成证据？

### 移动端检查

- 段落是否太长？
- 是否连续三行全是纯文字？
- 重点句是否足够突出？
- 表格是否适合手机阅读？

---

## 阶段 8：Final Package

最终交付不只是正文。

推荐交付包：

```text
candidate_links.md
source_log.md
user_voice_pack.md
core_conflict.md
visual_storyboard.md
article_framework.md
article.md
preview.md
review.md
images/
```

如果只交一个 Markdown，也至少包含：

- 选题判断
- 一手信源
- 热度依据
- 核心冲突
- 图文结构
- 正文
- 风险提醒
- 后续观察点

---

## 最终判断标准

一个合格选题包应该让编辑看完立刻知道：

1. 这个题为什么值得写
2. 事实从哪里来
3. 哪些说法不能写死
4. 用什么图证明
5. 文章从哪个冲突切入
6. 读者看完能获得什么

如果这六点不清楚，就还不是一个合格交付。
